# 项目4 · 智能客服 Agent（记忆 + 工具 + 人机协作）

> **综合实战**：构建一个带记忆、工具调用、情绪识别和安全防护的智能客服系统。

---

## TL;DR

> **30 秒速读**：构建一个带记忆、订单查询工具、情绪识别和 Prompt 注入防护的智能客服系统，能跨轮对话记住用户信息、检测负面情绪自动转人工。
> 
> **如果只记一件事**：情绪检测 + 自动转人工是客服 Agent 的安全阀，用户说"退款"/"投诉"时必须立刻切换到人工。

---

## 你会学到什么

1. **ConversationBuffer 记忆系统**：跨轮对话记忆，提取用户姓名和订单号
2. **订单查询工具**：从 `data/orders.json` 读取订单数据，按订单号查询状态
3. **情绪识别 + 转人工**：检测不满关键词（"投诉"/"退款"/"太差"等）→ 触发 Handoff
4. **防 Prompt 注入**：检测注入关键词（"忽略之前指令"/"admin"/"所有用户"等）→ 拦截
5. **离线 Mock**：API 不可用时自动降级为本地 fallback，`exit 0`

## 架构概览

```
用户输入
   │
   ▼
┌──────────────┐
│  安全检查    │──► Prompt 注入检测 → 拦截
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  情绪检测    │──► 负面情绪关键词 → 转人工
└──────┬───────┘
       │
       ▼
┌──────────────┐   query_order(orderId)
│  记忆 + 工具 │──► 提取用户信息 → 记忆更新
│              │──► 订单查询 → 格式化结果
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  LLM / Fallback │──► 生成回复
└──────────────┘
```

## 常见错误

> 概念懂了，实际写代码还是会踩坑。

| 错误 | 症状 | 解决 |
|------|------|------|
| 情绪检测只看单个关键词 | 用户说"服务态度太差了"没触发转人工 | 用多个同义词模式匹配，`太差\|很差\|不行\|垃圾\|失望` |
| 记忆系统没做截断 | 对话 50 轮后 context 窗口爆了 | 只保留最近 N 轮 + 关键信息摘要，或用 TokenBudget 限制 |
| 注入检测太敏感 | 用户问"你们的服务规则是什么"被拦 | 注入模式要精确，避免匹配正常问句里的"规则"/"指令"等词 |
| 订单查询没处理"不存在"的订单号 | 用户说"查 ORD-999"，Agent 返回空或报错 | 查询结果为空时返回"未找到该订单"，不要让 Agent 瞎编 |
| 转人工后没标记状态 | Agent 继续回复用户，和转人工逻辑冲突 | 设 `handoff_triggered` 标志，转人工后 Agent 停止回复 |

## 离线设计

`.env` 中 API 密钥为占位符 `sk-REPLACE-ME` 时：
- **订单查询**：纯本地 JSON 文件读取，不调 API
- **记忆系统**：纯本地数据结构，不调 API
- **情绪检测**：纯本地关键词匹配，不调 API
- **LLM 回复**：try/catch 失败 → 使用本地 fallback 回复

全程 **不依赖真实 API**，`exit 0`。

## 5 轮对话演示

| 轮次 | 用户输入 | 触发组件 |
|------|----------|----------|
| 1 | "你好，我叫张三" | 记忆：提取姓名 |
| 2 | "请帮我查一下订单 ORD-001 的状态" | 工具：订单查询 |
| 3 | "这个订单什么时候能到？" | 记忆：利用已知信息 |
| 4 | "等了这么久还没到，太差了！我要退款！" | 情绪：负面检测 + 转人工 |
| 5 | "算了，我要投诉你们的服务！" | 情绪：再次触发转人工 |
